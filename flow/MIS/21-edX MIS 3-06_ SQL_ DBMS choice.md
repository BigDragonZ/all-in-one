## 字幕内容

在之前的视频中，我们讨论了数据库管理系统（DBMS）如何通过将应用程序从数据管理任务中解放出来，从而简化其开发过程。这意味着应用程序和数据库管理系统需要商定一套指令或语言，应用程序将使用这套语言来请求数据库管理系统执行所有与数据相关的操作。

幸运的是，存在一个标准：SQL（Structured Query Language，结构化查询语言）。它是一种通用语言，通过它可以对数据进行定义、操作和查询。尽管每个品牌的数据库管理系统都有自己的一套指令，存在细微差别，但对于所有关系型数据库管理系统而言，SQL 的基本语法是相同的。因此，应用程序将使用 SQL 指令，或特定于 DBMS 的指令集，以及其自身编码所用的编程语言指令。

因此，一个应用程序需要通过某个给定的数据库管理系统来访问其数据时所发出的指令，与使用另一个不同的数据库管理系统时所需的指令非常相似。这有助于理解一个有趣的事实：软件公司在开发管理信息系统（作为计划销售给多个客户公司的软件包）时，通常会为多种数据库管理系统创建不同版本。

以复杂的集成企业管理信息系统（称为 ERP）为例，大多数 ERP 供应商（如 SAP）都支持多种关系型数据库管理系统的替代产品。为了更好地理解这一点，让我们再次审视我们简化的模型，该模型包含应用程序、数据库管理系统和数据库本身。在这种情况下，应用程序供应商（如 SAP）将开发和维护应用程序以及数据库的结构（或模型、或模式）。而数据库管理系统供应商（可能与应用程序供应商是同一家，也可能不是）则负责开发和维护 DBMS。为了简化流程，DBMS 的许可证有时会通过应用程序供应商进行销售。当然，数据库的实际数据内容将是用户公司的实际数据。

因此，如果一家公司想要购买并安装 SAP ERP 许可证，除了选择计算平台和操作系统外，还必须在 SAP 支持的数据库管理系统中进行选择。例如，如果该公司已经在使用 Oracle 的数据库管理系统来运行其他应用程序，它可能会选择使用同一个 DBMS 来运行 SAP。是的，尽管 SAP 和 Oracle 在 ERP 市场上是劲敌，但 SAP 的 ERP 确实支持 Oracle 的数据库管理系统。

另一方面，如果公司希望节省数据库管理系统的许可证费用，它可能会选择将其 SAP ERP 建立在其他成本较低的数据库管理系统之上，例如 Microsoft SQL Server 或 SAP 自己的 Sybase Adaptive Server Enterprise (ASE)。

总而言之，我们分析了数据不一致的问题，以及如何使用规范化来消除管理信息系统数据库中的冗余，从而消除不一致。我们讨论了 SQL，这是一种标准化的结构化语言，应用程序可以通过数据库管理系统使用它与数据库进行交互。我们还探讨了一些管理信息系统供应商如何允许其客户在多种数据库管理系统之间进行选择。
