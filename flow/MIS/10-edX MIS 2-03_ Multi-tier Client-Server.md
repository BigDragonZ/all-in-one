## 字幕内容

在之前的视频中，我们介绍了分布式信息系统以及更具体的客户端-服务器架构（Client-Server Architectures）的概念，并讨论了最简单的形式——双层客户端-服务器（two-tier client-server）。在本视频中，我们将扩展这一概念，讨论多层客户端-服务器架构。

从双层客户端-服务器模型的描述中可以看出，没有什么能阻止服务器再向层级更高的另一个服务器请求服务或数据，以此类推。这将形成一个具有多个层次（layers）、层级（tiers）或级别（levels）的客户端-服务器模型。

让我们来看一个具体的例子：一个集成信息系统，如 SAP 这样的 ERP（企业资源规划）系统。它是一个庞大而复杂的系统，用户通常通过运行在个人电脑（PC）上的客户端模块来访问。如果他们只是访问一台运行所有其他程序的中央计算机，这将是一个双层模型。然而，在大型部署中，用户通常会从客户端模块访问下一个层次，即第二层——应用层（application layer）。这一层包括构成信息系统的各种应用程序或模块的软件，例如销售模块、仓储模块或人力资源模块。这些不同的模块或应用程序可以驻留在一台或多台不同的服务器计算机上。它们中的每一个又会向下一层服务器（在此例中为第三层）发出数据或服务请求。在这个例子中，第三层将包含数据库。因此，这个三层客户端-服务器架构的例子包括：用户 PC 上的客户端模块向第二层的应用程序模块发出请求，而应用程序模块又会从位于系统核心的第三层公共数据库中请求数据。

如果我们想进一步扩展这个概念，可以看看那些以更“水平”方式与其他服务器通信的服务器。让我们讨论一下电子邮件系统的例子。在一个邮件系统中，你可以在你的客户端设备（PC、智能手机）上安装客户端模块，然后通过它访问你的邮件服务器。这个客户端模块负责与用户交互，也就是图形用户界面，并且它最终可能会在本地存储一些数据，比如一些邮件和照片。服务器模块则拥有收件箱，它会从其他服务器接收邮件，并转发你自己的邮件。然而，在这第一个服务器背后，显然会有一个由其他服务器组成的网络，负责在整个网络中交换邮件信息。

因此，我们可以看到，从严格的客户端-服务器概念到一个更分布式的架构，存在一个连续的谱系。另一个例子涉及通过网页浏览器进行访问。很多时候，我们不再使用专门的客户端模块，而只是在我们的客户端设备上使用网页浏览器，如 Chrome 或 Internet Explorer。使用网页浏览器，我们可以有一个简单的双层架构：如果网页浏览器只是通过互联网访问一个托管预存网页的 Web 服务器，浏览器请求页面，服务器提供页面，这就形成了一个双层模型。然而，如果我们想通过 Web 客户端访问一个应用程序（例如图书馆系统）来借书，我们的 Web 客户端会首先访问一个中间的 Web 服务器。该服务器会转换格式，理解请求内容，然后将具体的应用请求传递给一个应用服务器，该服务器包含了图书馆系统的逻辑和数据。如果我们请求借阅一本书，这个请求的结果会再次通过中间的 Web 服务器返回到 Web 客户端。这将是三层客户端-服务器架构的又一个例子。

在结束本节之前，我想强调，这些模型中的几种可以同时在同一个应用程序中共存，从而允许每个用户选择最方便的方式。例如，许多邮件系统可以直接从网页浏览器访问，但它们也为用户提供了在客户端设备上安装专用客户端模块的选项。

总而言之，我们可以看到，除了分布式系统的基础设施或通信基础设施，我们还可以有分布式信息系统。存在多种架构，其中最常见的是客户端-服务器架构。它可以以最简单的双层客户端-服务器模型存在，也可以以多层格式存在，并逐渐扩展或演变为一个完全的分布式架构。
